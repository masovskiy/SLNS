# SLNS - Sorry, Longer Not Supported

## О чём проект?

SLNS - это утилита для упаковки файлов в закрытый формат `.slns` с обязательным паролем.
У проекта одна задача: взять файл, закрыть его в контейнер и потом открыть обратно только по правильному паролю.
Команды у утилиты всего две:

```bash
snlsutil -p <file> <pass>
snlsutil -o <file> <pass>
```

`-p` — упаковать файл в `.slns`  
`-o` — открыть `.slns` обратно в исходный формат

SLNS не пытается быть архиватором на все случаи жизни. Он сделан под сценарий, где секретность для файла обязательна.

## Что мы используем?

Проект написан на **V**.
Внутри используются:
- `AES-256-CTR` для шифрования данных
- `HMAC-SHA256` для проверки целостности
- `PBKDF2-SHA256` для вывода ключей из пароля
- собственный контейнер `.slns`
- битовое представление содержимого в формате `k1=true, k2=false, ...`
SLNS работает локально и не требует внешних сервисов.

## Пример контейнера `.slns`
Контейнер SLNS — это текстовый файл со служебными полями и битовым телом.
Упрощённый пример:

```text
# SLNS PACKG VERSION 2
define .slnk=8f9a12ab34cd56ef...
define .alg=aes-256-ctr+hmac-sha256
define .kdf=pbkdf2-sha256:120000
define .salt=Q2hhbmdlTWU
define .iv=U29tZUlWVmFsdWU
define .name=ZG9jdW1lbnQucGRm
define .size=1024
define .mac=ab12cd34ef56...

define pckg:
 <slns v2.0 secure=true encrypted=true bits=true>
 <k1=true, k2=false, k3=true, k4=false, ...>
 <end>
pckg.v=true
```

Что здесь находится:

- `.slnk` — внутренний идентификатор контейнера
- `.alg` — алгоритм шифрования и защиты целостности
- `.kdf` — схема вывода ключей из пароля
- `.salt` — соль для пароля
- `.iv` — вектор инициализации
- `.name` — исходное имя файла в контейнере
- `.size` — исходный размер файла
- `.mac` — контроль целостности
- `k1=true, k2=false, ...` — тело контейнера, в котором лежат биты уже зашифрованного содержимого

## Что такое битовое представление?

После шифрования SLNS не хранит полезную нагрузку как бинарный блок.  
Он раскладывает каждый байт на 8 бит и записывает их в текстовом виде:

```text
k1=true, k2=false, k3=false, k4=true, ...
```

Например, если байт равен `01000001`, он будет записан как:

```text
k1=false, k2=true, k3=false, k4=false, k5=false, k6=false, k7=false, k8=true
```
То есть контейнер хранит не “символы файла”, а последовательность битов зашифрованного содержимого.

## Почему именно битовое представление?
Потому что для SLNS важно, чтобы контейнер был:
- полностью текстовым
- детерминированным
- разложимым на простые атомарные элементы
- легко проверяемым при чтении и восстановлении
- визуально отличимым от обычного архива или дампа base64
  
Важно: битовое представление само по себе не является главным источником защиты.  
Главную защиту дают шифрование, пароль и контроль целостности.

Битовое представление здесь нужно для другого:

- оно делает формат единообразным
- позволяет строго пересобирать данные обратно по количеству битов
- упрощает контроль структуры контейнера
- подчёркивает, что `.slns` — это собственный формат, а не просто ещё один архив

## Почему мы лучшие в информационной безопасности?

Если говорить именно о сценарии, где файл **обязан** быть закрыт по умолчанию, SLNS сильнее обычного хранения и лучше подходит для этой задачи, чем универсальные аналоги.

Почему:

- обычный файл вообще не предполагает секретность
- универсальные архиваторы делают секретность одной из опций
- SLNS строится вокруг секретности с самого начала
- пароль в SLNS не дополнительная функция, а базовое условие всей работы
- у SLNS только один понятный сценарий, а значит меньше лишней логики и меньше шансов на ошибку в использовании

Проще говоря:

- если секретность не нужна, файл можно хранить как есть
- если нужен универсальный архиватор, можно взять 7z
- если секретность обязательна и это главное требование, SLNS подходит лучше

## Сборка

Если у Вас установлен [V](https://vlang.io/), собрать утилиту можно так:

```bash
v -o snlsutil main.v
```

После этого можно использовать:

```bash
./snlsutil -p "document.pdf" "my-password"
./snlsutil -o "document.slns" "my-password"
```
